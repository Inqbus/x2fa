### 18. Nonce Replay Blocks Legitimate Reuse

| Field | Value |
|-------|-------|
| **File** | `src/x2fa/oidc/grants.py:95-100` |
| **Severity** | **MEDIUM** |

**Problem:**  
Nonce replay check searches `AuthorizationCode` rows. If an auth code expires (60s TTL) and is not exchanged, the nonce remains in the DB and blocks future logins with the same nonce.

```python
def exists_nonce(self, nonce, request):
    stmt = select(AuthorizationCode).where(AuthorizationCode.nonce == nonce)
    return g.db_session.execute(stmt).scalars().first() is not None
```

**Exploit Scenario:**  
User starts login → aborts → auth code expires → user starts login again → "Invalid nonce" because old nonce is still in DB.

**Fix:**  
Store nonces in a separate table with its own TTL (e.g., 1 hour), independent of AuthorizationCode lifetime. Or: mark nonce as used only when auth code is successfully exchanged.

```python
# New table: nonce_replay_protection
CREATE TABLE nonce_replay (
    nonce TEXT PRIMARY KEY,
    user_id INTEGER NOT NULL,
    expires_at TIMESTAMPTZ NOT NULL
);
-- Cleanup job removes expired nonces
```

### 19. Rate Limit per-IP, Not per-User

| Field | Value |
|-------|-------|
| **File** | `src/x2fa/routes/backup.py` + `src/x2fa/init_app/limiter.py` |
| **Severity** | **MEDIUM** |

**Problem:**  
Rate limit for backup code verification is `3/min; 10/hour`, but the key is `get_remote_address` (IP-based). An attacker in the same network (e.g., public WiFi) can attack backup codes of multiple users.

**Fix:**  
Implement per-user rate limit for backup code verification:

```python
# limiter.py — specific key for backup endpoint
@limiter.limit("3/minute; 10/hour", key_func=lambda: request.args.get("user_id", get_remote_address()))
def verify_backup():
    ...
```

Or: per-user failure tracking with progressive delay after 2 failed attempts.

### 20. IP Spoofing Without Reverse Proxy

| Field | Value |
|-------|-------|
| **File** | `src/x2fa/routes/__init__.py:8-12` |
| **Severity** | **MEDIUM** |

**Problem:**  
`X-Forwarded-For` is used unfiltered. If the app runs **without** a reverse proxy, the client can spoof its IP.

```python
def client_ip() -> str:
    return request.environ.get("HTTP_X_FORWARDED_FOR", "").split(",")[0].strip() \
        or request.environ.get("REMOTE_ADDR", "unknown")
```

`ProxyFix` trusts 1 proxy layer (`x_for=1`). Without proxy → `X-Forwarded-For` is set by client → IP spoofing.

**Fix:**  
Clarify documentation: x2fa **must** run behind a reverse proxy (nginx/Caddy). Optional: only accept `X-Forwarded-For` if `REMOTE_ADDR` matches a known proxy IP.

### 21. JWKS Fetch — No Key-Type/Algorithm Validation

| Field | Value |
|-------|-------|
| **File** | `src/x2fa/oidc/grants.py:215-239` |
| **Severity** | **MEDIUM** |

**Problem:**  
When fetching the JWKS URI, the response is parsed without validating key type or algorithm. A malicious RP could register a JWKS URI that returns a key with a different algorithm.

```python
with urllib.request.urlopen(req, timeout=5, context=ctx) as resp:
    jwks = json.loads(resp.read())
```

The `kid` match provides partial protection, but there is no check that the key is actually ES256.

**Fix:**  
Validate key type and algorithm after JWKS fetch:

```python
for key in jwks.get("keys", []):
    if key.get("kty") != "EC" or key.get("alg") != "ES256":
        raise ValueError(f"Unsupported key type/algorithm: {key.get('kty')}/{key.get('alg')}")
```

### 22. No FK Constraints — Orphaned Records

| Field | Value |
|-------|-------|
| **File** | `src/x2fa/model/*.py` |
| **Severity** | **MEDIUM** |

**Problem:**  
No SQLAlchemy `ForeignKey` constraints or cascade deletes. Example:

```python
# model/totp.py — BackupCode.user_id references User, but no FK
class BackupCode(Base):
    user_id = Column(Integer, nullable=False)  # No ForeignKey()
```

Consequences:
- User deletion does not cascade → orphaned records in Challenge, Credential, BackupCode, AuditLog
- No DB-level referential integrity
- Stale data about deleted users

**Fix:**  
Add FK constraints:

```python
user_id = Column(Integer, ForeignKey("user.id", ondelete="CASCADE"), nullable=False)
```

### 23. Rate Limit Storage: memory://

| Field | Value |
|-------|-------|
| **File** | `src/x2fa/init_app/limiter.py:3` |
| **Severity** | **LOW** |

**Problem:**  
`memory://` is per-worker. In multi-worker deployments (gunicorn with >1 worker), an attacker can bypass rate limits by hitting different workers.

**Fix for Production:**  
Use Redis or PostgreSQL as backend:

```python
limiter = Limiter(
    key_func=get_remote_address,
    storage_uri="redis://localhost:6379",
    storage_options={"url": "redis://localhost:6379"},
)
```

## Overall Assessment

**The x2fa server is production-ready.** The security implementation is well-designed and follows OIDC/WebAuthn best practices:

- PKCE S256 enforced at two levels
- Constant-time comparisons for all secret verification
- bcrypt with cost factor 12 for backup codes
- Fernet encryption for all sensitive fields at rest
- Comprehensive CSP with per-request nonces
- Correct WebAuthn configuration (user verification, origin, sign count)
- Generic error messages with no information leakage
- SHA-256 + salt IP hashing for GDPR compliance

**Recommended priorities for MEDIUM findings:**

1. **Nonce Replay** (#18) — Functional bug, affects user experience
2. **FK Constraints** (#22) — Data governance, no direct security risk
3. **Rate Limit per-User** (#19) — Relevant when multiple users share the same network
4. **JWKS Validation** (#21) — Only relevant when private_key_jwt is used as auth method
5. **IP Spoofing** (#20) — Covered by reverse proxy requirement
